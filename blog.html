The reasoning behind the current portfolio.

01ArchitecturePortfolio Site

Why the site needed more than one homepage.

A single page was useful at the beginning because it made progress visible quickly. But as projects, coursework, notes, personal direction, and contact information grew, every section began competing for the same attention. The homepage became a storage area instead of a clear entry point.

The solution was to assign one responsibility to each page. Home introduces the brand. Portfolio explains a few representative projects in depth. Practice Lab preserves the wider learning history. Blog records decisions. About connects manufacturing experience, quality habits, and the current software direction. Contact exposes only verified public paths.

Decision rule
  • Home is a gateway, not a database.
  • Portfolio prioritizes depth.
  • Lab preserves breadth and traceability.
02FrontendQA

Why the final page transition is intentionally simple.

The first goal was a cinematic transition between static HTML pages. Multiple versions were tested: a full-screen dark cover, native cross-document transitions, and stronger blur-and-scale effects. Each version solved one problem but created another, including long loading perception, white flashes, and unstable rendering on the classroom computer.

The final choice was a short, non-blocking micro-transition. It is less decorative, but it preserves continuity without hiding the page behind a loading screen. The lesson was that an effect is not successful because it looks impressive in isolation; it is successful when it remains stable in the real environment.

What was protected
  • No black loading hold.
  • No white flash.
  • No unstyled intermediate frame.
03AI WorkflowProject Governance

A practical division of responsibility between human, ChatGPT, and Codex.

Because I am still strengthening my software architecture and implementation fundamentals, vague requests can easily create misunderstanding, unnecessary token use, or code that is difficult to review. The workflow therefore starts by translating the goal into scope, constraints, allowed files, validation steps, and stop conditions before execution begins.

ChatGPT is used for planning, requirement translation, review, and risk checks. Codex is used for bounded repository execution. I run the result locally, compare it with the intended behavior, and approve or reject the next step. This does not remove the need to learn engineering; it creates a controlled way to build while making the missing knowledge visible.

Workflow sequence
  • Define and constrain.
  • Execute a small task packet.
  • Build and test locally.
  • Review evidence before approval.
